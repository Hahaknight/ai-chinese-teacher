# AI 语文教学助手小程序第一期开发计划

> 交接对象：Claude Code  
> 项目类型：微信小程序 + 后端服务  
> 第一期开工具功能：作文批改、病句修改、作文素材 / 小众意象库、讲评课生成器  
> 关键要求：底部导航极简；报告必须支持 Word / PDF 导出，方便老师打印发给学生。

---

## 0. 开发前置说明

当前已经完成网页版作文批改应用，但网页版操作不够便利。现在需要开发一个微信小程序版本。

小程序不是简单复刻网页版，而是围绕老师手机端真实使用场景重新设计：

- 老师用手机拍作文图片。
- 每个学生作文可以拍 1～3 页。
- 老师输入学生姓名，不再依赖文件名分组。
- 系统进行 AI 批改。
- 批改结果需要生成 Word / PDF，方便打印发给学生。
- 小程序同时集成几个轻量教学工具。

## 1. 产品定位

产品名称暂定：

```text
AI 语文教学助手
```

产品定位：

```text
面向语文老师 / 教培老师的 AI 教学工具箱小程序。
第一期围绕作文教学，完成批改、病句修改、作文素材查询、讲评课生成四个核心场景。
```

第一期核心价值：

```text
老师可以用手机完成作文采集和 AI 批改，并导出 Word / PDF 报告用于打印。
批改完成后，还可以基于全班作文结果生成讲评课方案。
```

## 2. 第一期开通功能范围

第一期只做 4 个工具：

```text
1. 作文批改
2. 病句修改
3. 作文素材 / 小众意象库
4. 讲评课生成器
```

第一期不做：

```text
1. 班级管理
2. 学生长期档案
3. 家长端
4. 支付系统
5. 多老师协作
6. 作文排行榜
7. PPT 自动生成
8. 作文错题本
9. 复杂后台管理系统
10. 小程序内给学生直接查看报告
```

原因：

```text
现在的学生在学校不允许使用手机，所以第一期的核心交付物不是手机端分享，而是老师导出 Word / PDF 后打印。
```

## 5. 推荐技术架构

### 5.1 小程序前端

推荐：

```text
微信小程序原生开发 + TypeScript
```

UI 组件库可选：

```text
TDesign Miniprogram
或 Vant Weapp
```

如果项目未来需要同时发布 H5 / App / 抖音小程序，可以考虑：

```text
uni-app
```

但第一期如果只做微信小程序，优先建议使用微信小程序原生开发，稳定、轻量、兼容性好。

---

### 5.2 后端

建议复用现有网页版作文批改应用的后端能力。

如果现有后端是 Next.js / Node.js，可以继续保留：

```text
小程序前端
→ 后端 API
→ 图片存储
→ AI 服务
→ 报告生成服务
→ 数据库
```

后端建议能力：

```text
1. 图片上传
2. AI 批改调用
3. 异步任务处理
4. 作文报告生成
5. Word / PDF 文件生成
6. 文件存储和下载链接生成
```

---

### 5.3 图片和文档存储

建议使用对象存储：

```text
腾讯云 COS
或阿里云 OSS
或 S3 兼容存储
```

需要存储：

```text
作文原图
作文报告 Word
作文报告 PDF
讲评课 Word
讲评课 PDF
```

---

### 5.4 数据库

建议继续使用现有数据库。

如果当前没有数据库，推荐：

```text
PostgreSQL + Prisma
```

或者：

```text
MySQL + Prisma
```

第一期不要只靠本地 JSON 文件，因为要支持历史记录、报告文件管理和导出文档管理。

---

### 5.5 AI 模型

作文批改需要视觉模型：

```text
支持图片输入的大模型
```

病句修改、讲评课生成可以使用文本模型。

作文素材库第一期建议静态内容为主，不依赖 AI 实时生成。

## 6. 功能一：作文批改

作文批改是第一期核心功能。

### 6.1 使用流程

```text
首页
→ 作文批改
→ 新建批改批次
→ 输入批改要求
→ 添加学生作文
→ 给每个学生拍摄 / 上传 1～3 张作文图片
→ 批量提交批改
→ 查看批改日志
→ 查看单篇报告
→ 导出 Word / PDF
```

---

### 6.2 新建批改批次页

字段：

```text
批次名称：必填
批改要求：必填
```

批次名称示例：

```text
五年级上册第一次作文批改
《那一刻，我长大了》作文批改
期中作文批改
```

批改要求输入框说明：

```text
请输入作文主题，也可以输入评分标准。

示例：
作文主题：那一刻，我长大了

评分标准：
1. 事件完整，能体现成长过程。
2. 有细节描写和心理描写。
3. 结构完整，结尾有感悟。
4. 满分 50 分。
```

规则：

```text
1. 批改要求只有一个多行输入框。
2. 用户可以只写作文主题。
3. 用户也可以同时写作文主题和评分标准。
4. 如果没有明确评分标准，系统使用默认 50 分标准。
5. 如果有明确评分标准，AI 优先按照用户输入的评分标准批改。
```

---

### 6.3 添加学生作文页

这是小程序版的关键页面。

页面标题：

```text
添加学生作文
```

页面结构：

```text
学生信息
- 学生姓名：必填

作文图片
- 第 1 页
- 第 2 页
- 第 3 页

操作按钮
- 拍照上传
- 从相册选择
- 保存并继续
- 保存并新增下一位
```

规则：

```text
1. 每个学生最少 1 张图片。
2. 每个学生最多 3 张图片。
3. 支持拍照上传。
4. 支持从相册选择。
5. 支持删除某一页图片。
6. 支持调整图片顺序。
7. 保存时必须填写学生姓名。
8. 小程序版不使用“文件名自动分组”。
9. 老师通过“学生姓名 + 作文图片卡片”管理每个学生的作文。
```

页面交互建议：

```text
1. 第 1 页、第 2 页、第 3 页以卡片形式展示。
2. 已上传的图片显示缩略图。
3. 未上传的页显示“上传第 X 页”。
4. 图片右上角提供删除按钮。
5. 下方显示当前顺序，支持拖拽排序或上移/下移。
6. 页面底部固定两个按钮：
   - 保存并继续
   - 保存并新增下一位
```

---

### 6.4 批次学生列表页

添加学生后，回到批次详情页。

展示内容：

```text
批次名称
批改要求摘要

学生作文列表：
- 张三  2 页  待批改
- 李四  1 页  待批改
- 王五  3 页  待批改

底部按钮：
- 添加学生作文
- 开始批量批改
```

规则：

```text
1. 可以继续新增学生。
2. 可以编辑已添加学生作文。
3. 可以删除某个学生任务。
4. 开始批改前，需要检查是否至少有 1 个学生作文。
```

---

### 6.5 批改任务处理

点击“开始批量批改”后：

```text
1. 创建 EssayBatch 批次任务。
2. 为每个学生创建 EssayTask。
3. 上传图片到对象存储。
4. 每个学生任务独立调用视觉模型。
5. 单个学生失败不能影响其他学生继续批改。
6. 每个任务需要更新状态。
7. 批改完成后生成 Word / PDF 报告。
8. 前端通过轮询或 websocket 查询任务状态。
```

第一期推荐使用轮询：

```text
每 3～5 秒请求一次任务状态。
```

任务状态：

```text
pending      等待处理
processing   批改中
success      批改成功
failed       批改失败
```

批次状态：

```text
pending      等待开始
processing   批改中
completed    全部成功
partial      部分成功，部分失败
failed       全部失败
```

### 6.8 Word / PDF 导出要求

这是第一期硬性需求。

必须支持：

```text
1. 单篇作文报告导出 Word。
2. 单篇作文报告导出 PDF。
3. 讲评课方案导出 Word。
4. 讲评课方案导出 PDF。
```

如果开发时间允许，增加：

```text
批量导出全部成功作文报告
```

批量导出格式：

```text
ZIP 压缩包
├── 张三_作文批改报告.docx
├── 李四_作文批改报告.docx
└── 王五_作文批改报告.docx
```

报告样式要求：

```text
1. A4 页面。
2. 黑白打印友好。
3. 不使用大面积彩色背景。
4. 标题清晰。
5. 分区明确。
6. 字号适合学生阅读。
7. 表格清楚。
8. 适合直接打印发给学生。
```

Word 生成建议：

```text
Node.js 使用 docx 包生成 .docx 文件。
```

PDF 生成建议：

```text
方案 A：HTML 模板 → Playwright / Puppeteer → PDF
方案 B：DOCX → LibreOffice 转 PDF
```

注意：

```text
小程序端不负责生成 Word / PDF。
Word / PDF 必须由后端生成，然后返回下载链接。
```

## 7. 功能二：病句修改

### 7.1 使用流程

```text
首页
→ 病句修改
→ 输入病句
→ 点击开始修改
→ AI 返回分析和修改建议
→ 一键复制
→ 保存记录
```

---

### 7.2 页面结构

输入区：

```text
请输入需要修改的句子
```

示例 placeholder：

```text
例如：我今天非常特别开心，因为妈妈给我买了一个礼物。
```

按钮：

```text
开始检查
```

输出区：

```text
原句
问题分析
修改版本
讲解说明
同类示例
```

输出示例：

```text
原句：
我今天非常特别开心，因为妈妈给我买了一个礼物。

问题分析：
“非常”和“特别”意思重复，两个程度副词同时使用会显得啰嗦。

修改版本：
我今天非常开心，因为妈妈给我买了一个礼物。

讲解说明：
表达情绪时，程度副词不要堆叠使用。选择一个即可，让句子更简洁自然。

同类示例：
错误：他跑得飞快极了。
修改：他跑得飞快。
```

---

### 7.3 病句修改验收标准

必须实现：

```text
1. 支持文本输入。
2. 调用 AI 返回病句分析。
3. 返回修改版本。
4. 返回讲解说明。
5. 返回同类示例。
6. 支持一键复制结果。
7. 保存最近修改记录。
```

暂不实现：

```text
1. 拍照识别病句。
2. 批量病句修改。
3. 病句专项训练题库。
```

## 8. 功能三：作文素材 / 小众意象库

注意：这里使用“小众意象库”，不是“小众意向库”。

### 8.1 功能定位

帮助老师快速找到可讲解、可迁移的作文素材和小众意象。

使用场景：

```text
1. 老师备课时找素材。
2. 老师讲作文时找示范片段。
3. 老师给学生提供更高级的写作表达。
4. 老师在批改后推荐学生使用某类素材。
```

---

### 8.2 第一期开通方式

第一期使用静态内置素材库，不依赖 AI 实时生成。

原因：

```text
素材库质量很重要，第一期应优先保证内容稳定、准确、可控。
```

第一期建议内置：

```text
30 个小众意象
20 个常见作文主题素材
```

总量约 50 条。

---

### 8.3 每条素材字段

```text
名称
分类
标签
核心特点
引申含义
适用主题
使用角度
示范片段
适合题目
避坑提醒
```

示例：

```text
名称：竹子

核心特点：
前几年生长缓慢，主要向地下扎根，后期快速生长。

引申含义：
厚积薄发、长期积累、默默蓄力、等待爆发。

适用主题：
成长、坚持、努力、积累、突破、自我改变。

使用角度：
可以写一个人长期努力看似没有结果，但其实一直在积累，最终在某个节点突然进步。

示范片段：
妈妈总说我像家里的那盆文竹，长了很久还是小小一株。可我知道，真正的竹子在地下悄悄扎根。那些做错的题、背过的书、流过的汗，都是我的根。直到那次考试，我第一次冲进班级前十，我知道，我的“第五年”到了。

适合题目：
《那一刻，我长大了》
《坚持让我更优秀》
《藏在时间里的答案》

避坑提醒：
不要只写“竹子很坚强”，要把竹子的生长特点和自己的经历结合起来。
```

## 9. 功能四：讲评课生成器

### 9.1 功能定位

讲评课生成器不是孤立工具，而是作文批改之后的延伸。

它要解决的问题：

```text
老师批改完一批作文后，需要快速整理出一节作文讲评课。
```

---

### 9.2 使用流程

```text
首页
→ 讲评课生成器
→ 选择已完成的作文批改批次
→ AI 读取该批次所有作文批改结果
→ 生成讲评课方案
→ 查看 / 编辑 / 保存
→ 导出 Word / PDF
```

---

### 9.3 输入来源

讲评课生成器不重新识别作文图片。

它直接读取作文批改结果：

```text
批改要求
学生人数
分数分布
每篇作文总评
每篇作文亮点
每篇作文主要问题
每篇作文修改建议
每篇作文简短评语
```

---

### 9.4 输出结构

```text
作文讲评课方案

一、本次作文整体情况
- 作文主题
- 批改人数
- 整体表现
- 分数分布

二、本次作文的主要优点
- 优点 1
- 优点 2
- 优点 3

三、本次作文的共性问题
- 问题 1
- 问题 2
- 问题 3

四、典型问题讲解
- 问题表现
- 原因分析
- 修改方法

五、优秀表达赏析
- 优秀开头
- 优秀结尾
- 细节描写
- 情感表达

六、课堂修改练习
```

## 13. AI Prompt 设计

### 13.1 作文批改系统 Prompt

```text
你是一名经验丰富的语文作文批改老师。

用户会上传一篇学生手写作文的 1 到 3 张图片，这些图片是同一篇作文的连续页面。你需要按照图片顺序阅读作文内容，并完成作文批改。

你会收到一个“批改要求”字段。这个字段可能只包含作文主题，也可能同时包含作文主题和评分标准。

你需要遵守以下规则：

1. 如果批改要求中包含明确评分标准，则优先按照用户提供的评分标准批改。
2. 如果批改要求中没有明确评分标准，则使用默认 50 分制：
   - 立意中心：10 分
   - 内容材料：15 分
   - 结构层次：10 分
   - 语言表达：10 分
   - 卷面与规范：5 分
3. 必须先识别图片中的作文文字。
4. 必须判断图片内容是否可读。
5. 如果图片模糊、缺失严重、无法识别正文，必须返回失败原因。
6. 不要虚构图片中不存在的内容。
7. 多张图片必须按照传入顺序理解为连续页面。
8. 批改要具体，不能只说“不错”“继续努力”。
9. 修改建议要可执行。
10. 改良后的作文必须保留学生原本的主要事件、情感和主题，不要完全重写成另一篇作文。
11. 输出必须是严格 JSON，不要输出 Markdown，不要输出解释性文字。
```

---
